## Authentication

Requires a valid authentication token in the request headers.

## Overview

Cultus uses a bi-directional sync protocol to support offline-first applications. Clients can:
1. **Pull** changes from the server to get updates since their last sync
2. **Push** local changes to the server to propagate offline modifications

This enables seamless synchronization across devices while allowing offline bookmark management.

## Query Parameters

<ParamField query="last_synced_at" type="string" required>
  ISO 8601 timestamp of the client's last successful sync. Server returns all changes that occurred after this timestamp.
</ParamField>

## Response

Returns a changeset containing created, updated, and deleted bookmarks since the last sync.

<ResponseField name="changes" type="object" required>
  Container for all database changes
  <Expandable>
    <ResponseField name="bookmarks" type="object">
      Bookmark table changes
      <Expandable>
        <ResponseField name="created" type="array">
          Array of newly created bookmark objects
        </ResponseField>
        <ResponseField name="updated" type="array">
          Array of updated bookmark objects (full objects, not diffs)
        </ResponseField>
        <ResponseField name="deleted" type="array">

## Sync Flow

1. Client stores the timestamp of last successful sync
2. Client calls `GET /user/sync?last_synced_at=<timestamp>`
3. Server returns all changes since that timestamp
4. Client applies changes to local database
5. Client stores the new timestamp from response
6. Repeat on next sync

## Push Changes to Server

<api>POST /user/sync</api>

Push local bookmark changes to the server, typically created or modified while offline.

## Side Effects

- Created bookmarks trigger background processing for link metadata extraction
- New bookmarks publish events to the queue system
- The server assigns permanent IDs to locally created bookmarks

## Empty Sync

If `changes.bookmarks` is null or `timestamp` is 0, the request succeeds with a "no changes to sync" message.

## Conflict Resolution

The server uses a "last write wins" strategy. The client is responsible for:
- Detecting conflicts based on `updated_at` timestamps
- Merging changes intelligently when conflicts occur
- Preserving user intent in conflict scenarios

## Integration with WatermelonDB

The sync protocol is designed to work seamlessly with [WatermelonDB](https://nozbe.github.io/WatermelonDB/), a reactive database for React Native:

- Field names use `_created_at` and `_updated_at` to avoid WatermelonDB's protected fields
- The changeset format matches WatermelonDB's sync adapter expectations
- Local IDs allow tracking bookmarks before server assignment
